# Quickstart: Host-Side IMS Registration over LTE

**Feature**: `015-volte-host-ims` | **Date**: 2026-07-22

How to reproduce the verified hardware state, run the blocking Gate G1 spike,
and validate the feature once built.

## Hardware baseline

| Item | Value |
|---|---|
| Module | Quectel EC200U, firmware `EC200UCNAAR03A12M08` (UNISOC UIS8910) |
| Carrier | Vodafone India (Vi), MCC 404 / MNC 43 |
| Radio | LTE FDD Band 3 |
| AT port | `/dev/ttyUSB0` (`ttyUSB1`–`ttyUSB3` are not AT; `ttyUSB4`–`6` unprobed) |
| Data path | CDC-ECM, host interface `enx024bb3b9ebe5` |
| Control plane | **AT only** — no QMI, no MBIM, no `/dev/cdc-wdm*` |

Interface names are host-specific; confirm before use.

## Step 0 — Verify the modem is attached

```bash
lsusb | grep 2c7c                 # expect 2c7c:0901
ls /dev/ttyUSB*                   # expect ttyUSB0..6
ip -br link | grep enx            # find the ECM interface
```

Then, on `/dev/ttyUSB0` at 115200:

```
ATE0
AT+CPIN?              -> +CPIN: READY
AT+COPS?              -> +COPS: 0,0,"Vi India",7
AT+CEREG?             -> +CEREG: 0,1        (registered on LTE)
AT+QCFG="ims"         -> +QCFG: "ims",2,0   (modem's own IMS idle — not contending)
AT+CIREG?             -> +CIREG: 0,0
```

If `AT+CIREG?` reports a registered state, the modem's internal IMS stack is
active and will contend for the IMS PDN. Resolve that before continuing.

## Step 1 — Bring up the IMS PDN (verified working)

This transcript was executed successfully against the live network.

```
AT+CGDCONT=3,"IPV4V6","ims"
AT+CGACT=1,3
AT+CGACT?                -> +CGACT: 3,1
AT+CGCONTRDP=3           -> apn resolves to ims.mnc043.mcc404.gprs, bearer id 6
AT+CGPADDR=3             -> +CGPADDR: 3,"0.0.0.0,2402:8100:...:3801"
AT+QNETDEVCTL=1,3,1
AT+QNETDEVCTL?           -> +QNETDEVCTL: 1,3,1,1
```

The host ECM interface should transition from `DOWN`/`NO-CARRIER` to `UP`.

**Expect an IPv6-only address.** The `0.0.0.0` IPv4 entry is normal.

**This re-points the modem's single host data path from `internet` to `ims`.**
On the reference machine that costs nothing (the interface was unused; the
host routes via `enxa84a63261317`). Confirm the same before running it
elsewhere. To restore: `AT+QNETDEVCTL=1,1,1`.

## Step 2 — Route the host onto the IMS PDN *(verified working)*

Requires `CAP_NET_ADMIN`. **A plain Claude Code / shell session on this host
cannot do this** — `sudo` is blocked by `no_new_privileges`. Use the container:

```bash
docker run --rm --privileged --network host --entrypoint sh \
  docker-gsm-sip-bridge:latest -c '...'
# tools not in the image: apk add --no-cache tcpdump ndisc6 dhcpcd
```

### The critical step: adopt the modem-assigned IID

**The PDN is unusable until you do this.** The network unicasts its Router
Advertisement to the link-local form of the IID it assigned — *not* to
`ff02::1` — so Linux's autogenerated stable-privacy link-local causes every RA
to be silently discarded. Symptoms are `no IPv6 Routers available` and
`Address not available`, with the RAs arriving the whole time.

Take the IID from `AT+CGPADDR=3` (e.g. `…:0:C:DE2B:3801` → `fe80::c:de2b:3801`):

```bash
IF=enx024bb3b9ebe5
LL=fe80::c:de2b:3801        # derive from CGPADDR — changes on every reactivation

ip link set $IF down
ip -6 addr flush dev $IF
echo 1 > /proc/sys/net/ipv6/conf/$IF/addr_gen_mode     # 1 = none
ip link set $IF up
ip -6 addr add $LL/64 dev $IF scope link
sysctl -w net.ipv6.conf.$IF.accept_ra=2
rdisc6 -1 $IF
```

Expected result:

```
inet6 2402:8100:6ffe:8ae6:4b:b3ff:feb9:ebe5/64 scope global dynamic proto kernel_ra
default via fe80::c:de2b:3840 dev enx024bb3b9ebe5 proto ra
```

The host is now routed on the carrier's IMS PDN.

> **Open question (R9)**: SLAAC derives the global address from the ECM MAC
> (EUI-64), not from the modem-assigned IID. 3GPP expects the assigned IID.
> Whether the network routes the EUI-64 address is unverified — add the
> `CGPADDR` address explicitly and prefer it as the SIP source until proven
> otherwise.

## Step 2b — P-CSCF discovery: Gate G1 **already executed, result negative**

Do **not** re-run this expecting success. Recorded here so the negative is
reproducible and so a future firmware or carrier can be re-tested cheaply.

```bash
# DHCPv6 Information-Request with ORO for options 21,22 (RFC 3319)
python3 dhcp6.py $IF     # see research.md R2
# -> reply from fe80::5, msg-type 7, DNS = ['::','::'], NO option 21/22

dig +short NAPTR ims.mnc043.mcc404.3gppnetwork.org   # no resolver: DNS is ::
```

| Method | Result |
|---|---|
| DHCPv6 | Server exists and replies; returns **no** SIP-server options |
| RA | Prefix + MTU only; **no** P-CSCF option |
| DNS | Unusable — only `::` offered as resolver |
| `AT+QCFG="pdn/pco",1` | Accepted, persists across reboot, **never populates** |

**Supply the P-CSCF via `--pcscf` / config instead.** See plan.md Gate G3 for
how to obtain an address.

## Step 3 — Gate G2: Gm IPsec over IPv6

Verify ESP transport-mode SAs install with IPv6 selectors, **before**
attempting registration, so a failure here is not misread as a registration
failure:

```bash
ip xfrm state add src <local-v6> dst <pcscf-v6> proto esp spi 0x00000101 \
   mode transport auth hmac(md5) 0x<key> enc cipher_null ""
ip xfrm policy add src <local-v6> dst <pcscf-v6> dir out \
   tmpl proto esp mode transport
ip xfrm state list && ip xfrm policy list
ip xfrm state flush && ip xfrm policy flush     # clean up
```

The SIP layer is already IPv6-clean (research R3); this is the one untested
v6 surface.

## Step 4 — Validate the feature

Once implemented:

```bash
# US1 — attachment
gsm-sip-bridge volte-pdn --action up
#   expect: assigned APN ims.mnc043.mcc404.gprs, bearer 6, IPv6, host iface named
gsm-sip-bridge volte-pdn --action up          # idempotent: reports reuse, exit 0
gsm-sip-bridge volte-pdn --action status

# US2 — discovery
gsm-sip-bridge volte-discover
#   expect: per-method report, and which method won
gsm-sip-bridge volte-discover --method dhcpv6  # isolate a single method

# US3 — registration
gsm-sip-bridge volte-register --once
#   expect: accepted registration, or a named failing stage

# US4 — lifecycle
gsm-sip-bridge volte-register                  # stays up, renews
gsm-sip-bridge volte-status

# teardown restores the previous data-path binding
gsm-sip-bridge volte-pdn --action down
```

### Success criteria mapping

| Check | Criterion |
|---|---|
| `volte-pdn --action up` succeeds in <60s, no hand-entered address | SC-001 |
| `volte-discover` finds the P-CSCF and names the method | SC-002 |
| `volte-register --once` accepted in <60s | SC-003 |
| Registration survives two renewals | SC-004 |
| Each induced failure names its stage | SC-005 |
| VoWiFi suite passes unchanged **and a live VoWiFi call completes** | SC-006 |
| `volte-status` shows registration state | SC-008 |

## Non-regression (SC-006) — mandatory

The VoWiFi path is in production. Before any commit touching `ims/`:


`ImsTransport` refactor (Phase 1) touches shared code, so a green suite alone
is not sufficient evidence.

## Troubleshooting

| Symptom | Likely cause | Action |
|---|---|---|
| `AT+CGACT=1,3` returns ERROR | Network refused the IMS PDN, or context 3 already in use | Check `AT+CGACT?`; try a different cid |
| `CGCONTRDP` shows an empty assigned APN | PDN not genuinely granted | Confirm bearer id differs from the internet context's |
| Host interface stays `DOWN` after `QNETDEVCTL` | Bind did not take | Check `AT+QNETDEVCTL?` reports `1,3,1,1` |
| No IPv6 global address | RA not received/accepted | `sysctl net.ipv6.conf.$IF.accept_ra` must be `2` with forwarding on |
| `sudo: no new privileges` | Not in the privileged container | Run inside compose, not on the sandboxed host |
| `+CME ERROR: 58` on a PCO query | Firmware does not support it | Expected on EC200U — this is why Gate G1 exists |
| Modem's IMS stack contends | `AT+QCFG="ims"` enabled | Set idle; confirm `+CIREG: 0,0` |
